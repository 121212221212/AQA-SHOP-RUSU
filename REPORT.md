##  Решённые технические проблемы


###  Проблема 1: Docker vs Standalone
**Проблема:** Изначально проект предполагал использование Docker, но наше устройство не поддерживало вриртуализацию

**Решение:**
- Использована standalone версия AQA Shop (aqa-shop.jar)
- Настроено прямое подключение к localhost:8080
- Переписаны тесты под реальное приложение
- Вместо Docker контейнера открыть (http://localhost:8080) 



###  Проблема 2: Селекторы кнопок
**Проблема:** Кнопки "Купить" и "Купить в кредит" имели одинаковые CSS классы

**Решение:**
- создание дополнительного теста по поиску селекторов кнопок и полей заполнения
- Использованы селекторы по тексту вместо CSS классов
- Реализованы надёжные XPath селекторы



###  Проблема 3: Настройка Gradle зависимостей

**Проблема:** Конфликты версий и отсутствующие зависимости

**Решение:**

- Чистая настройка build.gradle.kts

- Явное указание совместимых версий

- Добавление недостающих зависимостей (SLF4J, Allure)



### Проблема 4: Запуск AQA Shop

**Исходная задача:** Развертывание AQA Shop в Docker контейнере согласно техническому заданию

**Проблема:**
- Устройство не поддерживает виртуализацию для Docker

**Исследование решения:**

1. **Поиск альтернатив** - найдена standalone версия AQA Shop (aqa-shop.jar)
2. **Сравнение функциональности** - убедились что функциональность идентична

**Реализованное решение:**

Вместо Docker контейнера использовали прямое подключение
open("http://localhost:8080");



### Проблема 5: Allure отчёты

**Проблема:**  Allure serve зависал на 66% на Windows 

**Решение:**

- Использование статических отчётов вместо serve

- Генерация HTML отчёта с последующим ручным открытием



### Проблема 6: Запуск тестов через терминал

**Проблема:** Тесты работали через IDEA, но падали при запуске через терминал

**Решение:**

- Фокусировка на работоспособности через IDEA

- Документирование проблемы как известного ограничения

- Подготовка альтернативных отчётов


### Проблема 7: Архитектурная 

**Проблема:** Прямые вызовы Selenide в тестах приводили к дублированию кода


**Решение:** 

**Создание структуры**

- MainPage (Выделение MainPage для главной страницы)
- DebitPage (Выделение MainPage для главной страницы)
- CreditPage (Создание CreditPage для оплаты к кредит)

**Преимущества:**

- Повторное использование кода

- Упрощение поддержки

- Читаемость тестов
